# TRDD-9c392c3b — AMAMA v2.10 follow-up hardening (deferred #12 items)

## ⏵ STATE — READ THIS FIRST ON RESUME — 2026-06-13

This TRDD captures the **#12 items intentionally NOT shipped in v2.10.0**, with
exact specs, so v2.10.0 ships the governance-correctness + memory-migration core
and these are tracked rather than rushed. Two classes: **DETERMINATE** (do
anytime — content is known) and **MODEL-DEPENDENT** (BLOCKED until the canonical
AI Maestro governance/auth model — `team-governance/references/GOVERNANCE-RULES.md`
or equivalent — is consulted; guessing would recreate the very contradictions
#12 is closing).

### DETERMINATE (content known — execute when picked up)
- **M7 — loop-ownership statements.** Add to `skills/amama-amcos-coordination/SKILL.md`:
  the ORCHESTRATOR owns the design→dispatch handshake; the INTEGRATOR owns the
  transition INTO `complete`/`published`/`live`; no MEMBER self-marks a TRDD
  `complete`. (Grounded in the R6/workflow model + `manager-approval-defaults.md`.)
- **M10 — inbox priority enforcement.** Add to `skills/amama-user-communication/SKILL.md`:
  process unread AI Maestro messages **URGENT > HIGH > NORMAL** before resuming the
  current task (the AI MAESTRO MESSAGING priority rule — already in CLAUDE.md, not yet
  in a skill).
- **M5-doc (optional, low value).** The substance of M5 already ships as PRRD **S9.1**
  (MANAGER cannot self-approve its own releases → USER approves). A doc-form restatement
  in `skills/amama-approval-workflows/SKILL.md` is a nice-to-have, not required.

### MODEL-DEPENDENT (BLOCKED on the canonical governance/auth model — do NOT guess)
- **M11b-1 — `docs/ROLE_BOUNDARIES.md` "3-Role System" vs "8 titles".** The doc
  accurately describes the TEAM-role axis (manager/COS/member); the #12 flag is the
  collision with the governance-TITLE axis (MANAGER/AUTONOMOUS/MAINTAINER + the member
  specializations). Reconcile by clearly labelling the two axes — but confirm the exact
  title set against the canonical source first.
- **M11b-2 — `docs/FULL_PROJECT_WORKFLOW.md` "Manager assigns COS" (line 10).** #12 says
  this is WRONG (COS is not assigned by MANAGER in the current model). Determine who
  assigns COS from the canonical model before editing — `ROLE_BOUNDARIES.md` currently
  agrees with the "wrong" version (line 23), so BOTH may need the same correction.
- **M11b-3 — `skills/amama-approval-workflows/references/governance-password.md` vs
  AID_AUTH-only.** #12 says the standalone governance-password contradicts an
  AID_AUTH-only policy. The file describes a SERVER-side GovernanceRequest password
  (distinct from `prrd-edit.py`'s `$AID_AUTH` caller check). Confirm whether the server
  still uses a password or is AID_AUTH-only, then update or delete (no legacy versions).
- **amama-session-memory reconciliation.** `skills/amama-session-memory/SKILL.md` is a
  THIRD memory mechanism (CozoDB + `thoughts/shared/handoffs/amama/` + "write handoffs
  before session end"). It is SEPARATE from the retired activeContext bank and from the
  new three-scope markdown system. Decide: superseded by the three-scope model (retire
  it) or complementary (keep, but rewire its "before session end" step — the session_end
  HOOK was deleted in `9a53977`; only the agent-invoked skill path remains). Needs a
  janitor-#18-aligned decision.

### LARGER FOLLOW-UPS (own effort; not restart-blocking)
- **M9 — approval-tier watchdog (D3/D4 implementation).** Today the watchdog is
  DOC-ONLY (`trdd-approval-tiers.md` Part D). Implement the objective tier-floor
  classifier + the lazy idle-sweep auditor (raise under-classified TRDDs, move
  self-approved-but-should-be-gated back to `design/proposals/`) as a real script +
  tests. The approval-tier system already works MANUALLY via `amama_proposal_approvals.py`,
  so this is enforcement hardening, not a gate.
- **broad M12 — test coverage.** Only `amama_proposal_approvals.py` + the memory system
  have real tests. Add real tests for every hook (session_start, user_prompt_submit,
  stop_check), every skill, every command. Dispatch `python-test-writer` agents with an
  EXPLICIT per-target test count (else it defaults to ~30/function).

structure (PRRD + 4-zone folders + approval-tiers), the memory-system migration
(old bank retired, three-scope adopted), and R6-v3 routing. The items above are
either low-priority-determinate, model-dependent (unsafe to guess), or larger
enforcement/test efforts — correctly sequenced AFTER the core ships.
